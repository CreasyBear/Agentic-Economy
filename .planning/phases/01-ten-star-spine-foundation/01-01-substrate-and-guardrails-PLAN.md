# 01-01 — Substrate and Guardrails Plan

## Objective

Create the smallest real TanStack Start + Convex + Clerk runtime substrate and fail-first guardrails that stop backup coupling, future-surface drift, unsafe TypeScript, UI drift, and public overclaim before any Phase 1 behavior is built.

## Authority Inputs

- `01-SPEC.md` R1, R10 and prohibitions.
- `01-CONTEXT.md` decisions D-1, D-2, D-4, D-5, D-26 through D-34.
- `01-RESEARCH.md` sequencing recommendation for `01-01-substrate-and-guardrails`.
- `01-PATTERNS.md` clusters 1, 2, 12, 13, 16.
- `01-VALIDATION.md` Wave 0 requirements.
- `01-UI-SPEC.md`, `DESIGN.md`, `.impeccable/design.json`, and `.planning/FRONTEND-DESIGN-FRAMEWORK.md` for the first UI substrate.
- Skills: `tanstack-start-best-practices`, `tanstack-router-best-practices`, `clerk-tanstack-patterns`, `convex`, `convex-setup-auth`, `shadcn`, `accessibility`, `playwright-best-practices`, `product-design`.

## Scope

### In

- Package and app scaffold only after verifying current package names/install commands from official docs or package metadata during execution.
- `package.json`, strict TypeScript config, Vite/TanStack Start app shell, Convex folder, Playwright/Vitest configs, and empty source/test directories.
- Official shadcn initialization only after runtime substrate exists; install the smallest primitive set needed by the UI framework seed.
- `src/styles/tokens.css`, `src/styles/globals.css`, `src/components/ui/*`, `src/components/ae/layout/*`, `src/components/ae/status/*`, `src/lib/ui/{status-presentation,copy,contract-scans}.ts` with no real claim/publish/domain behavior.
- Fail-first scanner tests and bad fixtures for imports, source-mining, TypeScript standards, copy, and UI contract.

### Out

- No claim/publish, registry, admin, discovery, API, payment, protected action, inbox, notification, or public business behavior.
- No fake routes beyond a non-mutating shell proof route required to compile/render the design substrate.
- No package version guesses in docs or code comments; execution verifies packages first.

## Product Design Pass

- **Primary user/job/object/outcome:** future owner/operator needs a trustworthy foundation; object is the runtime spine; outcome is executable guardrails before any user-facing capability can lie.
- **Reachable states in this slice:** scaffold compiles, scanner passes clean tree, scanner fails bad fixtures, non-mutating shell renders if needed, command failure shows actionable test output.
- **Design constraints:** `DESIGN.md` is the visual seed; `FRONTEND-DESIGN-FRAMEWORK.md` owns tokens/components; no route may ship bespoke shadcn defaults as final UI.
- **Rendered proof:** if a shell route exists, capture compact and wide screenshots only as framework evidence, not as claim/publish product proof.

Also create and run explicit negative-fixture scripts so each scanner is observed failing for the intended fixture without polluting the clean command: `npm run test:imports:fixtures`, `npm run test:source-mining:fixtures`, `npm run test:ts-standards:fixtures`, `npm run test:copy:fixtures`, and `npm run test:ui-contract:fixtures`.

## Stop Conditions

- Any package cannot be verified from current official docs or package metadata.
- Any guardrail can only pass by becoming a no-op.
- Any runtime file imports from `../Agentic-Economy-Backup`, `.planning`, or module `internal/*` across seams.
- Any user-facing route claims a Phase 1 behavior that is not implemented in this PR.
